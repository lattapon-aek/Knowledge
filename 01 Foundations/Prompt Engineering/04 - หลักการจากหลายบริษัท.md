# หลักการจากหลายบริษัท



---

## หลักการที่ตรงกันจากหลายบริษัท

### 1. กำหนดเป้าหมายให้ชัด

- **Microsoft** — specific ลดพื้นที่การตีความ
- **Google Cloud** — กำหนด objective และ expected outcomes ก่อนทดสอบ
- **Anthropic** — เริ่มจาก success criteria ก่อนปรับ prompt

> prompt ควรตอบได้ว่า "ให้ทำอะไร", "ให้ทำกับข้อมูลไหน", "ผลลัพธ์หน้าตาแบบไหน", "อะไรถือว่าผิด"

---

### 2. ใส่บริบทที่จำเป็น

- OpenAI + Google Cloud — ใส่ context ที่เกี่ยวข้องกับงาน
- AWS — บริบทเป็นองค์ประกอบหลักของ prompt
- ถ้าคำตอบต้องอิงเอกสาร/ข้อมูลเฉพาะ → แนบในตัว prompt หรือใช้ RAG

---

### 3. ใช้ตัวอย่างเมื่อจำเป็น

- few-shot/multi-shot รองรับจาก OpenAI, Google Cloud, AWS, Anthropic
- เหมาะเมื่อต้องการควบคุม style, format, decision boundary, edge case

---

### 4. จัดโครงสร้างให้โมเดลอ่านง่าย

- **OpenAI** — แบ่งบทบาทและแยก role guidance ออกจาก task details
- **Anthropic** — แนะนำ XML tags
- **Google Cloud** — เน้น ordering, labeling, delimiters
- **Microsoft** — "order matters"

---

### 5. ระบุข้อจำกัดและรูปแบบผลลัพธ์

- **Microsoft** — เจาะจงมากที่สุด
- **OpenAI** — structured outputs / schema เมื่อเหมาะสม

---

### 6. ทำงานแบบ Iterative และใช้ Eval

- **Google Cloud** — test-driven และ iterative
- **Anthropic** — วัดผลเชิงประจักษ์
- **OpenAI** — eval, prompt versioning, prompt reuse

> อย่าตัดสิน prompt จากตัวอย่างเดียว ทดสอบกับชุดกรณีที่สะท้อนงานจริง

---

### 7. งานใหญ่ควรแตกเป็นขั้น

- **Microsoft** — LLM ทำงานดีขึ้นเมื่อแตก task เป็น smaller steps
- **Anthropic** — chain complex prompts
- **Google Cloud** — ถ้า prompt ซับซ้อนเกินไป ควร simplify หรือ restructure

---

### 8. Parameter มีผล แต่ไม่แทน Prompt ที่ดี

- **Microsoft** — temperature และ top_p มีผล ควรปรับทีละตัว ไม่ใช่พร้อมกัน
- **Google Cloud** — ทดลอง parameter เป็นส่วนหนึ่งของการปรับ

> ถ้า prompt ยังไม่ชัด การไปปรับพารามิเตอร์อย่างเดียวมักไม่แก้ต้นเหตุ

---

## Best Practices จาก Microsoft

1. **Be Specific** — ระบุให้ชัด
2. **Be Descriptive** — อธิบายให้ครบ
3. **Double Down** — ย้ำสิ่งสำคัญ
4. **Order Matters** — ลำดับมีผล
5. **Give the model an "out"** — ให้โมเดลมีทางออกเมื่อข้อมูลไม่พอ

---

## Best Practices จาก Anthropic

1. กำหนด success criteria ก่อนเขียน prompt
2. เทคนิคเรียงจาก broadly effective → specialized: clarity, examples, XML tags, role prompting, thinking, prompt chaining, long context handling
3. ถ้าปัญหามาจาก latency/cost → เปลี่ยน model ดีกว่าปรับ prompt

> "อย่าเริ่มแต่ง prompt ก่อนรู้ว่าอะไรคือคำตอบที่ดี"

---

## Best Practices จาก Google Cloud

- Prompt engineering เป็น test-driven และ iterative
- ประสิทธิภาพขึ้นกับทั้ง **content** และ **structure**
- ถ้าผลไม่ดี อย่ารีบเพิ่มความยาว บางครั้งการจัดเรียงใหม่ช่วยมากกว่า
- Multimodal prompt อาจตอบต่างกันเมื่อสลับลำดับระหว่างไฟล์กับคำสั่ง
